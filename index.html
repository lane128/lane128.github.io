Fastlane+RVM使用小结

RVM

  1. 什么是RVM？
    Ruby enVironment (Version) Manager. 它是一个命令行工具，可以提供一个便捷的多版本 Ruby 环境的管理和切换。

  2. RVM的基本使用。

    • 查看可安装的ruby版本

      rvm list known

    • 安装指定版本的ruby

      rvm install x.x.x

    • 查看已经安装过的ruby版本

      rvm list

    • 使用指定版本的ruby作为现在环境的ruby版本 (如果添加 --default 的flag，那么下次启动终端将自动使用该版本作为ruby的默认版本)

      rvm use x.x.x --default

    • 卸载已经安装过的ruby版本

      rvm remove x.x.x

  3. RVM的Gemset使用

    RVM不单单为一个环境提供了不同ruby版本共存的环境，还可以提供不同ruby环境下 gem 环境的的配置，这就是gemset。一个特定的gemset可以视为一个独立虚拟的 Gem 环境。 在一个特定ruby版本下的 gemset 下可以安装不同的libs来构建需要的环境。

    建立一个ruby版本是2.3.0，gemset 名字为 testset 的 Gem 环境。

    rvm install 2.3.0 #假如 rvm 没有安装2.3.0 ruby

    rvm use 2.3.0 #切换到ruby为2.3.0的版本环境

    rvm gemset create testset #在当前的ruby环境下建立一个名字为 testset 的gemset环境。

    => ruby-2.3.0 - #gemset created /Users/lane/.rvm/gems/ruby-2.3.0@testset

    => ruby-2.3.0 - #generating testset wrappers........

    rvm use 2.3.0@testset #使用 ruby 为 2.3.0下的 testset 的gemset

    => Using /Users/lane/.rvm/gems/ruby-2.3.0 with gemset testset

    现在你就拥有一个 2.3.0 版本 ruby 的 gem 环境了。更多的内容参考 RVM官网

Fastlane 中 Match 的使用。

  1. Fastlane 的安装

    官方是推荐使用 Gemfile 去定义 fastlane 的依赖库。利用 bundler 来快速安装 fastlane。细节请参考这里.

  2. Match 的使用

    Match - A new approach to code signing. 是一个用来方便管理你项目的 certificates 和 provisioning profiles 的命令工具。方便的在不同的机器上快速部署这些东西。具体的信息可以参考官网介绍.

遇到的问题

  1. 由于 Match 是使用私人的 git 库来储存加密过的 certificates 和 provisioning profiles 文件。
    所以更新或者新配置环境都是需要通过 git 来获取的。 在 macOS 10.12 版本后，需要重新把 rsa 私钥文件添加到 ssh-add 中去。不然在访问 git 的时候会要求你输入 passphrase 参考这里. fastlane 的 2.4.0 版本没有对这个进行处理，会导致在 Match 获取 git 内容的时候卡住在访问 git 库上。

  2. 在用 Match 配置好 development 、 adhoc 、 appstore 后，出现 development 的 provisioning profiles 出现与证书不匹配的情况， 请在系统证书中删除并不是当前项目相关的那个证书即可。

Objective-C Naming

在项目中，正确地为class, method, function, constants 等命名是有助于代码的review以及代码可扩展的重要条件。

一般原则

声明

  • 如果能做到简洁明了的命名是最好，但是不应该为了“简洁”而牺牲“明了”

    clarity shouldn’t suffer because of brevity

Code Commentary
insertObject:atIndex: Good.
insert:at: Not clear; what is being inserted? what does “at” signify?
removeObjectAtIndex: Good.
removeObject: Good, because it removes object referred to in argument.
remove: Not clear; what is being removed?
  • 一般来说不要说些某些名字，把它拼写完整，即使它很长，因为不同的开发者有着不同的文化和语言背景，往往你可能觉得很好理解的一个缩写，其它开发者可能会confused。所以尽量避免一些不是真的被广泛和长久使用过的缩写。可以参考 Acceptable Abbreviations and Acronyms。
Code Commentary
destinationSelection Good
destSel Not clear
setBackgroundColor: Good
setBkgdColor: Not clear
  • 避免语意模棱两可的API名字，这种命名容易被不同的方式理解。
Code Commentary
sendPort Does it send the port or return it?
displayName Does it display a name or return the receiver’s title in the user interface?

一致性

  • 在不同类里的做同一件事情的方法应该有一样的名字。特别在Objective-C中如果利用到多态性，那么更要保持一致性原则。
Code Commentary
- (NSInteger)tag Defined in NSView, NSCell, NSControl.
- (void)setStringValue:(NSString *) Defined in a number of Cocoa classes.

不自我引用

  • 命名不要自我引用，这样显得多余。这就和简洁性原则相违背。
Code Commentary
NSString *dataString Okay
NSString *dataStringObject Self-referential

例外，当常量是掩码(Mask)/)类型或者是通知(Notification)名为例外。

命名前缀

都知道苹果为自己的framework定义了前缀来区分不同framework和第三方开发者的类库。这种命名前缀有特定的格式，一般都是以2-3个大写的字母(不包含下划线)来做前缀。

Prefix Framework
NS Foundation
NS Application Kit
AB Address Book
IB Interface Builder
AF AFNetworking(第三方)

排版上的习惯

一般来说Objective-C多用驼峰命名规范(Camel-Casing)，第一个单词首字母小写，后面的字幕首字母都是大写。

对于方法名一般为驼峰命名：fileExistsAtPath:isDirectory:

对于功能和常量名，一般添加前缀并把单词的首字母都大写: NSRunAlertPanel NSCellDisabled

应该避免在方法前面添加下划线来表示这是一个私有方法，Apple保留了这一协议给自己的私有变量命名。比如你定义的`@property (nonatomic, strong) NSString myProperty*变量会对应生成 *_myProperty`*。如果引入第三方库，有可能第三方库会不经意的用它自己的私有方法修改一个已经存在的私有方法。这样引起的后果很严重。

类与协议的命名

类的命名应该包含一个能很好地解释这个类的作用的名词，名字前面应该有适当的前缀。这种例子很多，在Foundation 和 Application framework里特别多。

NSString NSDate UIButton

很多协议没有与特定的类关联，这种类型的协议应该用动词ing形式来命名。

Code Commentary
NSLocking Good
NSLock Poor (seems like a name for a class)

头文件

  • 声明一个孤立的类或者协议
Header file Declares
NSLocale.h The NSLocale class
  • 声明一个关联类和协议
Header file Declares
NSString.h NSString and NSMutableString classes
NSLock.h NSLocking protocol and NSLock, NSConditionLock, and NSRecursiveLock classes
  • 引入framework头文件
Header file Declares
Foundation.h Foundation.framework
